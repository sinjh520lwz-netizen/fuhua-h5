+# 第一章：回测系统基础架构与设计哲学

+## 1.1 什么是回测？为什么回测不等于真实交易？
+
+回测（Backtesting）的核心定义：**用历史数据模拟策略的买卖过程，统计盈亏表现**。
+
+很多初学者把回测想得太简单了——"不就是看过去的数据，按规则买卖，算算赚了多少吗？"实际上，一个**可信的**回测系统需要处理的问题远比想象中复杂：
+
+**问题一：你真的能以那个价格成交吗？**
+
+假设你的策略在2026年3月15日发出信号"买入600519贵州茅台，价格1800元"。但那天茅台开盘直接跳空高开到1850元，你的1800元挂单根本不会成交。如果你的回测系统用1800元作为买入价，你的收益就被高估了。
+
+**问题二：涨跌停板导致你买不到或卖不掉**
+
+A股有10%的涨跌停限制（科创板/创业板20%）。如果一只股票涨停了，你想买入但排队买单排了几百万手，你根本买不到。如果你的回测系统在涨停价标记"买入成功"，结果就是假的。
+
+**问题三：T+1制度**
+
+A股今天买的股票，明天才能卖出。如果你的回测系统允许今天买今天卖，那就完全脱离了现实。
+
+**问题四：手续费和滑点**
+
+每次交易都有佣金（万分之二到万分之三）、印花税（卖出千分之一）、过户费（万分之零点二）。加上实际成交价和你看到的价格之间总有几跳的差距（滑点）。这些"小钱"累积起来非常可观——假设你每天交易一次，每次千分之三的成本，一年250个交易日就是75%的手续费。
+
+**问题五：幸存者偏差**
+
+你用今天还在交易的3024只股票来回测，但过去10年里有几百只股票退市了。如果你的回测不包含那些退市的股票（它们大概率是亏损股），你的策略表现就会被系统性高估。
+
+**问题六：未来函数（Look-Ahead Bias）**
+
+这是最容易犯也最致命的错误。比如用当天收盘价来决定当天是否买入——实际上你在收盘前不可能知道收盘价。或者用全市场平均市盈率来筛选个股——这个平均值包含了你正在评估的那只股票。
+
+**本章核心原则：一个好的回测系统，必须让回测环境尽可能接近真实交易环境。宁可低估收益，也不要高估。**
+
+## 1.2 回测系统的两大架构
+
+### 架构一：事件驱动型（Event-Driven）
+
+```
+历史数据 → 逐根K线推送 → 策略引擎处理 → 交易引擎模拟 → 统计模块汇总
+```
+
+每一根K线到达时，策略引擎收到一个事件（on_bar），根据当前持仓状态和市场数据决定是否下单。交易引擎接收订单，检查是否能成交（价格、涨跌停、资金），更新持仓。统计模块记录每一笔交易的细节。
+
+**优点**：最接近真实交易流程，能精确模拟下单-撮合-成交的每一步。
+**缺点**：代码复杂度高，运行速度慢（需要逐根K线遍历）。
+
+**适用场景**：需要精确模拟盘中行为（如分钟级策略、条件单策略）、需要模拟订单簿深度。
+
+### 架构二：向量化型（Vectorized）
+
+```
+历史数据 → 批量计算信号 → 批量计算收益 → 统计汇总
+```
